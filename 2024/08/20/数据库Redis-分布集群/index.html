redis-cluster集群概述

1.集群是什么

cluser是一个redis实现分布式管理数据的功能

Redis Cluster 是 Redis 分布式解决方案，它可以让我们将 Redis 数据库分布在多个节点上，实现数据的高可用性和扩展性。简单来说，就是把一个大的 Redis 数据库拆分成多个小的部分，并将这些部分分布在不同的服务器上，这些服务器被称为节点。

每个节点都存储着部分数据，并且节点之间通过主从复制来保持数据的一致性。当有新的节点加入集群或者节点出现故障时，Redis Cluster 会自动进行数据的重新分配和故障转移，确保数据的可用性和系统的稳定性。

通过使用 Redis Cluster，我们可以横向扩展 Redis 数据库的容量和性能，以应对大规模数据和高并发请求的需求。同时，它还提供了简单易用的管理工具和命令，方便我们对集群进行配置和监控。

总结起来，Redis Cluster 就是一个可以将 Redis 数据库分布在多个节点上，并提供高可用性和可扩展性的解决方案，帮助我们构建分布式的 Redis 服务。

分布式系统的含义

分布式是指将一个系统中的不同组件或节点分布在多台独立的物理或虚拟机器上，通过网络通信协作来完成特定的任务或提供特定的服务的架构设计方式。在分布式系统中，各个组件或节点可以同时运行，并通过网络通信来实现数据共享、负载均衡、容错处理、高可用性等功能。

分布式系统通常用于解决单个计算机无法满足性能需求或容量需求的情况，通过将系统拆分成多个独立的部分并分布在多个节点上，可以更有效地利用资源、提高系统的可靠性和灵活性。分布式系统可以应对大规模并发访问、海量数据处理、高可用性要求等复杂的场景，是现代大型网络应用、云计算、大数据处理等领域的重要技术基础。

系统是什么意思

上文中，“系统”通常指的是一个或多个相互关联的组件、服务或软件，以及它们之间的交互关系，共同完成特定功能或提供特定服务的整体。在技术领域中，系统可以指代一个软件系统，如操作系统、数据库系统、网络系统等，也可以指代一个硬件系统，如计算机系统、服务器系统等。

在上文中提到的分布式系统，指的是将多个系统组件或节点分布在多台独立的物理或虚拟机器上，通过网络通信协作来完成特定任务或提供特定服务的整体架构。这些系统组件可以是软件模块、服务实例、数据库节点等，它们通过网络通信协作来实现数据共享、负载均衡、容错处理等功能。

横向扩展的含义

横向扩展是指通过增加系统中的多个实例或节点来提高系统的容量和性能。横向扩展也称为水平扩展，与纵向扩展相对应。

纵向扩展是通过增加单个实例的资源（如增加内存、处理器等）来提高系统的性能和容量，但存在着硬件限制和成本较高的缺点。而横向扩展则是通过增加多个相同或相似的实例或节点来提高系统的性能和容量，可以更灵活地应对增加的负载和数据量，且可以通过添加更多的节点来实现无限的扩展。

横向扩展通常用于分布式系统中，例如通过增加服务器节点来扩展数据库、缓存或应用服务的能力。通过横向扩展，系统可以更容易地实现高可用性、负载均衡和弹性扩展，从而满足不断增长的用户需求和数据规模。

2.集群的作用

cluster作用

  1. 横向扩展和高可用性：Redis Cluster 允许将数据分片并分布到多个节点上，实现横向扩展和高可用性，通过添加更多节点来扩展系统容量和吞吐量。
  2. 负载均衡：Redis Cluster 自动将数据分布到不同节点上，实现负载均衡，提高系统性能和可扩展性。
  3. 自动故障转移：当主节点发生故障时，Redis Cluster 支持自动故障转移，选举出新主节点，确保系统可用性。
  4. 分片管理：Redis Cluster 负责将数据按规则分布到不同节点上，实现负载均衡和横向扩展，提高系统性能和可扩展性。

总之，Redis Cluster 可帮助构建具有横向扩展、高可用性、负载均衡和分片管理的分布式系统，适用于大规模数据处理和高并发访问场景。

4.集群对请求的处理

redis cluster采用去中心化的架构，集群的每一个节点都复制一部分槽位，客户端在写入key时，到底放到哪个槽位呢？

Redis 集群对请求的处理主要依赖于==重定向机制==

总的来说，节点对请求的处理过程大致如下：

  1. 节点首先会根据 CRC16 算法或其他哈希算法计算键的槽位，然后查询负责该槽位的节点。
  2. 如果槽位对应的节点不是当前节点，那么当前节点会返回一个特殊的错误响应，告诉客户端正确的节点地址，进行自动的重定向。
  3. 如果槽位对应的节点是当前节点，当前节点会立即处理该请求，返回槽位中键的结果。
  4. 如果槽位对应的节点是当前节点，但是键不在当前节点的槽位中，节点会检查槽位是否在迁移中。
  5. 如果槽位正在进行迁移，节点会返回一个特殊的错误响应（ASK 错误）给客户端，告诉客户端正确的节点地址，进行手动的重定向。
  6. 如果槽位不在迁移中，节点会检查槽位是否正在导入中。
  7. 如果槽位正在导入中，并且有 ASKING 标记，则节点会直接操作，而不返回重定向响应。
  8. 否则，节点会返回一个特殊的错误响应（MOVED 错误），告诉客户端正确的节点地址，进行自动的重定向。

集群重定向

在 Redis 集群中，当客户端发送一个需要访问集群数据的命令时，如果该命令需要操作的键所在的槽位不在当前节点上，那么该节点会返回一个特殊的错误响应，称为”MOVED”。这个响应会告诉客户端正确的节点地址。客户端会根据这个信息，重新向正确的节点发出请求。

另外，如果集群中的某个节点正在进行手动故障转移（manual failover）操作，那么在这个过程中，集群节点也会返回一个特殊的响应，称为”ASK”。这个响应也会包含正确的节点地址，客户端也会重新向正确的节点发出请求。

这种通过特殊的错误响应实现的重定向机制，可以确保客户端能够正确地访问集群中的数据，即使数据分布在不同的节点上。

5.集群的工作原理

Redis Cluster 是 Redis 提供的一个分布式解决方案，可以将数据分布在多个节点上，提高系统的性能和可用性。Redis Cluster 的工作原理主要包括以下几个方面：

  1. 数据分片：Redis Cluster 使用分片策略将数据分布在多个节点上，每个节点负责存储部分数据。分片策略通常是通过对数据的 key 进行哈希计算，然后将数据分配到不同的节点上。
  2. 节点间通信：Redis Cluster 中的各个节点会通过 Gossip 协议进行通信，维护集群的拓扑结构和节点状态信息。通过 Gossip 协议，集群中的节点可以相互发现、交换信息，确保集群中之间的通信和数据同步。
  3. 主从复制：每个分片在 Redis Cluster 中通常会有多个副本节点，其中一个是主节点，其余是从节点。主节点负责处理客户端的读写请求，而从节点用于备份和提供读取服务。当主节点故障时，从节点可以自动晋升为主节点。
  4. 故障检测和自动恢复：Redis Cluster 能够自动检测节点的故障，并在节点故障时进行自动故障转移。当主节点故障时，从节点可以自动晋升为主节点，继续提供服务。
  5. 客户端路由：客户端在向 Redis Cluster 发送请求时，会根据数据的 key 计算出对应的分片，然后通过槽位映射表将请求转发到对应的节点上。这样可以保证客户端的请求可以正确路由到对应的节点上，实现数据的读写操作。

总的来说，Redis Cluster 通过数据分片、节点间通信、主从复制、故障检测和自动恢复，以及客户端路由等技术实现了分布式部署和高可用性。这些机制保证了 Redis Cluster 能够在多节点环境下提供高性能、高可用性的服务。

原理补充

redis集群功能无法实现读写分离

  1. 读写分离和负载均衡：Redis集群中的分片技术主要用于实现数据的分布式存储和负载均衡。分片技术将数据分散存储在不同的主节点（master）中，以实现更大容量和更好的性能。然而，Redis集群并没有内置的读写分离功能，即读操作和写操作仍然需要直接连接到主节点处理。因此，读写操作并没有被分离到不同的节点上，这可能会导致主节点的负载过高。
  2. 主从复制和高可用性：Redis集群中的从节点（slave）主要用于实现数据的备份和故障切换的高可用性。当主节点出现故障或不可用时，从节点可以自动接管主节点的工作，确保数据的持久性和可用性。因此，从节点的存在不仅仅是为了实现故障切换，也可以提高系统的可用性和稳定性。

综上所述，Redis集群在分片技术方面主要用于负载均衡和数据分布，而主从复制则主要用于实现高可用性和数据备份。虽然Redis集群并没有实现读写分离和读操作的负载均衡功能，但通过合理的架构设计和配置，仍然可以满足大部分应用的需求。希望这些解释对您有所帮助。如果您有更多疑问，请随时提出。

6.集群节点相互通信的作用

在 Redis Cluster 中，每个节点需要相互通信的原因主要有以下几点：

  1. 集群管理和节点发现：Redis Cluster 是一个分布式系统，每个节点都需要与其他节点进行通信来协调集群管理和节点发现。通过相互通信，每个节点可以了解整个集群的拓扑结构，发现其他节点的位置和状态，并协调数据分片和节点故障转移等操作。
  2. 数据同步和复制：在 Redis Cluster 中，数据会被分片到不同的节点上，同时每个分片会有多个副本。节点之间需要相互通信来进行数据同步和复制，确保数据的一致性和可靠性。
  3. 故障检测和自动故障转移：通过相互通信，节点可以及时检测到其他节点的故障，并根据配置的故障转移机制自动进行数据迁移和重新分配，以确保系统的可用性和稳定性。
  4. 客户端请求路由：客户端请求会通过集群的负载均衡器或路由功能被路由到合适的节点上处理。节点之间需要相互通信来协调客户端请求的路由和数据传输，使整个集群能够高效地处理请求。

综上所述，Redis Cluster 中每个节点需要相互通信是为了协调集群管理、数据同步、故障转移和客户端请求路由等功能，确保整个集群的运行稳定和高效。

7.集群的请求入口设计

在Redis Cluster中没有一个单独的入口节点的设计，以及任意一个节点都可以连接到集群的设计，主要有以下几个原因：

  1. 高可用性和负载均衡：通过没有入口节点的设计，任意一个节点都可以充当客户端节点，避免了单点故障的风险。如果一个节点故障，客户端可以通过其他节点来访问集群，提高了系统的可用性。同时，这也实现了负载均衡，客户端请求可以分布到不同的节点上，避免了单个节点的性能瓶颈。
  2. 简化集群管理：没有入口节点的设计简化了集群的管理，不需要额外的配置或管理入口节点。每个节点都可以独立地加入或离开集群，降低了集群的维护成本。
  3. 灵活性和弹性：由于任意一个节点都可以连接到集群，这种设计提供了更大的灵活性和弹性。新的节点可以方便地加入集群，旧的节点可以方便地离开集群，而不会对整个系统造成影响。
  4. 提高性能：通过任意一个节点都可以访问到集群的方式，可以实现请求的并发处理和分布式存储，从而提高了系统的性能。

因此，Redis Cluster中没有入口节点且任意一个节点都可以连接到集群的设计，是为了提高系统的可用性、灵活性、性能和管理简单性。这种设计能够更好地满足分布式系统的需求。希望这些解释能够帮助理解为什么Redis Cluster采用这样的设计。如果还有其他问题，请随时提出。

8.集群总线端口概念

Redis 集群中的总线端口是用于节点之间进行通信和数据同步的端口。默认情况下，每个节点的总线端口号是其主实例端口号加上 10000。例如，如果主实例端口号是 6379，那么对应的总线端口号就是 16379。这个总线端口是用于集群节点之间进行通信、同步复制和数据交换的重要通道。

9.集群数据分片策略

数据分片策略是指在分布式系统中将数据分割成多个部分（片段）存储在不同的节点上的一种策略。在Redis集群中，数据分片策略用于将数据分布在不同的节点上，以实现负载均衡和提高系统性能和可用性。

在Redis集群中，数据分片策略通常采用哈希槽分片的方式。具体来说，Redis将所有的键通过哈希算法计算得到一个哈希槽（slot）的编号，然后根据这个编号将数据分配到不同的节点上。这样每个节点负责存储一部分数据，当需要查找或写入数据时，客户端需要根据哈希算法计算键对应的哈希槽，并通过集群的槽映射信息找到存储该槽的节点，然后向该节点发送请求。

通过数据分片策略，Redis集群可以实现数据的水平扩展，提高系统的吞吐量和容量。同时，数据分片策略也带来了一些挑战，比如数据重定向、跨节点事务处理等问题需要进行处理。

redis-cluster部署实战

  1. 检查节点状态：首先，redis-cli工具会检查指定的Redis节点的状态，包括节点是否已经在集群中、节点的数据是否为空等。
  2. 创建揭晓：然后，redis-cli会在指定的Redis节点上创建一个集群。这个过程包括为集群选择一个主节点（Master）和设置副本节点（Replica），并在集群中建立通信通道。
  3. 数据分片：接下来，redis-cli会根据一致性哈希算法将数据分片到不同的节点上。这样可以实现数据在集群中的分布，提高性能和扩展性。
  4. 同步数据：如果指定了副本数量，redis-cli会在集群中为每个主节点创建指定数量的副本。这些副本节点会定期从主节点同步数据，以实现数据的冗余备份。
  5. 更新集群状态：最后，redis-cli会更新集群的状态，并将配置信息写入各个节点的配置文件中。这样，整个集群就被成功创建了。

需要注意的是，redis-cli --cluster create命令是在Redis官方提供的redis-cli工具基础上执行的，通过该命令可以方便地创建Redis集群。在执行该命令之前，需要确保Redis节点已经正确配置，并且网络通信正常。

配置文件内添加参数

cluster-enabled yes
cluster-config-file nodes_6380.conf

  1. cluster-enabled yes：这个参数指示Redis是否启用了集群模式。当设置为yes时，表示Redis实例将以集群模式运行，这意味着您可以在Redis Cluster中使用多个节点来管理数据。
  2. cluster-config-file nodes_6380.conf：这个参数指定了Redis Cluster的配置文件路径。在Redis Cluster中，每个节点都会保存一个关于集群拓扑结构和其他信息的配置文件。通过设置这个参数，您可以指定配置文件的路径和名称。
  3. cluster-node-timeout 15000：这个参数指定了在Redis Cluster中节点之间进行通信时的超时时间，单位为毫秒。在Redis Cluster中，各个节点之间需要频繁地进行通信，比如进行故障检测、故障转移等操作。通过设置这个参数，您可以指定节点之间通信的超时时间，当超过这个时间后节点将被认为是不可达的。

通过Redis命令行工具redis-cli中的--cluster create命令实现自动化创建集群

这个过程包括节点发现、槽位分配、主从关系建立以及集群配置更新等步骤

通过这些操作，Redis 集群可以自动完成槽位分配和主从关系建立，从而简化了集群的部署过程，减少了手动操作的工作量

自动创建集群工作原理

==三步==1.发现集群节点、2.进行槽位分配、3.建立主从关系

使用 redis-cli --cluster create 命令自动部署 Redis 集群的工作原理如下：

  1. 节点发现：当执行 redis-cli --cluster create 命令时，redis-cli 会首先连接到指定的节点，这些节点将会成为 Redis 集群的初始节点。连接成功后，redis-cli 会向每个节点发送 CLUSTER MEET 命令，用于让节点相互发现和建立通信连接。
  2. 槽位分配：在集群中，数据会根据槽位进行分片存储，每个槽位对应一个哈希槽，这样可以将数据均匀地分布在集群的各个节点上。在执行 redis-cli --cluster create 命令时，redis-cli 会自动为每个节点分配一定数量的槽位，并创建槽位映射。
  3. 主从关系建立：根据指定的副本数量，redis-cli 会自动为每个主节点创建相应数量的从节点。主节点负责处理客户端请求和写入操作，而从节点则用于备份数据和提供读取服务。redis-cli 会自动向各个节点发送 CLUSTER REPLICATE 命令，建立主从关系。
  4. 集群配置更新：在执行 redis-cli --cluster create 命令的过程中，redis-cli 会向各个节点发送 CLUSTER ADDSLOTS 命令，用于更新节点的槽位信息，并向其他节点发送 CLUSTER REPLICATE 命令，更新节点的主从关系。
  5. 完成集群创建：当所有节点成功加入集群、槽位分配完成、主从关系建立完毕后，集群创建过程就完成了。此时，Redis 集群已经建立，可以开始对集群进行读写操作。

总的来说，redis-cli --cluster create 命令通过自动连接和管理各个节点，进行槽位分配和主从关系的建立，最终完成了 Redis 集群的自动部署过程。这样可以简化集群部署的过程，减少手动操作的工作量。

集群相关使用命令

  • 集群
    cluster info ：打印集群的信息
    cluster nodes ：列出集群当前已知的所有节点（ node），以及这些节点的相关信息。

  • 节点
    cluster meet ：将 ip 和 port 所指定的节点添加到集群当中，让它成为集群的一份子。
    cluster forget ：从集群中移除 node_id 指定的节点。
    cluster replicate ：将当前从节点设置为 node_id 指定的master节点的slave节点。只能针对slave节点操作。
    cluster saveconfig ：将节点的配置文件保存到硬盘里面。

cluster reset： 清空节点的集群环境数据

  • 槽(slot)
    cluster addslots [slot …] ：将一个或多个槽（ slot）指派（ assign）给当前节点。
    cluster delslots [slot …] ：移除一个或多个槽对当前节点的指派。
    cluster flushslots ：移除指派给当前节点的所有槽，让当前节点变成一个没有指派任何槽的节点。
    cluster setslot node ：将槽 slot 指派给 node_id 指定的节点，如果槽已经指派给
    另一个节点，那么先让另一个节点删除该槽>，然后再进行指派。
    cluster setslot migrating ：将本节点的槽 slot 迁移到 node_id 指定的节点中。
    cluster setslot importing ：从 node_id 指定的节点中导入槽 slot 到本节点。
    cluster setslot stable ：取消对槽 slot 的导入（ import）或者迁移（ migrate）。
    • 键
      cluster keyslot ：计算键 key 应该被放置在哪个槽上。
      cluster countkeysinslot ：返回槽 slot 目前包含的键值对数量。
      cluster getkeysinslot ：返回 count 个 slot 槽中的键 。

命令自动创建集群

redis-cli --cluster create 是用于在Redis集群中创建集群的命令

--cluster-replicas 1 表示集群中为每个主节点创建一个副本节点

-a password 创建集群连接实例节点的密码

架构关系如下，为什么要每个机器的master和另一个机器的slave建立主从，而不是和自己机器的slave建立关系

是因为防止机器故障，若主从关系是一个机器的两个节点配置，那机器挂了两个节点就挂了，无法进行高可用故障转移

挂掉的机器就会影响部分数据写入与读取，所以分开机器进行建立，一台机器彻底挂了，另一台机器依然还有一个节点存在

检查集群整体信息

cluster info 命令用于获取有关集群的整体信息，包括集群节点数量、槽位分配情况、复制和握手状态等。

参数解释

执行了 cluster info 命令，用于获取 Redis 集群的信息，输出结果解释如下：

  • cluster_state:ok：表示集群状态正常

  • cluster_slots_assigned:16384：表示总共有 16384 个槽位被分配

  • cluster_slots_ok:16384：表示所有槽位都正常

  • cluster_slots_pfail:0：表示没有槽位处于部分失败状态

  • cluster_slots_fail:0：表示没有槽位处于完全失败状态

  • cluster_known_nodes:6：表示集群中已知的节点数为 6 个

  • cluster_size:3：表示集群中有 3 个节点 （1主1从包括为1个节点）

  • cluster_current_epoch:6：表示当前集群的 epoch 为 6（表示集群每个节点状态都正常都为1）

  • cluster_my_epoch:1：表示当前节点的 epoch 为 1（表示当前实例节点的状态值，正常为1.默认为0）

  • cluster_stats_messages_ping_sent:4585：表示发送的 ping 消息数为 4585 条

  • cluster_stats_messages_pong_sent:4982：表示发送的 pong 消息数为 4982 条

  • cluster_stats_messages_sent:9567：表示总共发送的消息数为 9567 条

  • cluster_stats_messages_ping_received: 表示已接收到的 ping 消息的数量。Ping 消息通常用于检测节点的活跃状态，以及节点之间的通信延迟。

  • cluster_stats_messages_pong_received: 表示已接收到的 pong 消息的数量。Pong 消息是对 ping 消息的响应，用于确认节点的活跃状态和计算通信延迟。

  • cluster_stats_messages_meet_received: 表示已接收到的 meet 消息的数量。Meet 消息用于在 Redis 集群中发现新的节点。

  • cluster_stats_messages_received: 表示总共接收到的消息数量，包括以上提到的 ping、pong、meet 消息以及其他类型的消息。

检查节点详细信息

cluster nodes 命令用于获取有关集群节点的详细信息，包括节点 ID、地址、角色、主节点 ID、复制偏移量、最后一次同步时间、握手状态以及槽位分配情况。

cluster nodes 提供了更详细的节点级信息，而 cluster info 则提供了对整个集群的高级概览信息。

参数解释

  1. 第一列是节点的 ID，用于唯一标识每个节点。
  2. 第二列是节点的地址和端口，格式为 IP:端口@内部通信端口。
  3. 第三列是节点的角色，可能是主节点（master）或从节点（slave）。
  4. 如果节点是从节点，则第四列是它正在复制的主节点的 ID，如果是主节点那么后面就是 –
  5. 第五列是复制偏移量，指示从节点上次复制的位置。
  6. 第六列是最后一次同步时间的 UNIX 时间戳。
  7. 第七列是节点的握手状态，表示与其他节点的连接状态。
  8. 最后一列（可选）是节点负责的槽位范围，以表示集群中数据分片的分配情况。
  9. myself,slave 其中的myself是对自己的一个标识，当前是通过那个节点查询的集群信息，那个节点就会带有myself

测试集群分片策略

批量写入数据，测试分片数据策略是否会平均写入每个节点

小结：

通过测试得知，分片策略会将数据分片平均存储到各个对应的卡槽节点，这样的好处是

  1. 增加存储容量：通过将数据分散存储在多个节点上，可以有效地增加系统的存储容量。每个节点存储一部分数据，整个系统的存储容量就是各个节点存储容量的总和。
  2. 提高读写性能：数据分片可以使系统同时处理多个读写操作，从而提高系统的读写性能。每个节点只负责处理自己存储的数据，可以并行地执行读写操作，提高系统的吞吐量和响应速度。
  3. 实现负载均衡：将数据分片存储到不同节点上可以实现负载均衡。系统可以根据节点的负载情况，将请求路由到负载较轻的节点上进行处理，从而平衡系统的负载，提高系统的稳定性和性能。
  4. 提高可用性：分片策略可以增加系统的可用性。当某个节点发生故障时，其他节点仍然可以继续正常运行，保证系统的可用性。同时，可以通过数据备份和复制来提高数据的容错能力，防止数据丢失。
  5. 水平扩展：通过分片策略，系统可以实现水平扩展，即通过增加节点来扩展系统的容量和性能。当数据量增加时，可以动态地增加新的节点，将数据分片存储到新节点上，从而实现系统的横向扩展。

综上所述，通过将数据分片平均存储到各个对应的卡槽节点，分片策略可以增加存储容量、提高读写性能、实现负载均衡、提高可用性和实现水平扩展等好处。
