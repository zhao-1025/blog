redis主从复制概述

我们知道要避免单点故障，即保证高可用，便需要冗余（添加副本/salave节点）方式提供集群服务。

而Redis 提供了主从库模式，以保证数据副本的一致，主从库之间采用的是读写分离的方式。

redis的主从复制关系中从节点默认只能接收读的请求

1.主从复制概述

Redis主从复制是指在Redis中设置一个主服务器（master）和多个从服务器（slaves），主服务器将自身的数据复制到从服务器上。这样可以实现数据的备份和读写分离。

主从复制主要有以下几个特点：

  1. 数据备份：主从复制可以实现数据的备份和容灾，如果主服务器发生故障，从服务器可以接管服务，保证数据不丢失。（引入哨兵功能）
  2. 读写分离：主服务器负责处理写操作，而从服务器可以负责处理读操作，从而分担主服务器的负载。
  3. 扩展性：通过添加多个从服务器，可以扩展系统的读取能力，提高系统的并发处理能力。

2.主从复制作用

  1. 数据冗余和备份：通过主从复制，可以在从节点上创建数据的冗余备份，以防止主节点发生故障时造成数据丢失。
  2. 负载均衡：主从复制可以分担主节点的读取负载，从而提高系统的读取性能。读取请求可以分发到从节点上，减轻主节点的压力。
  3. 故障恢复：如果主节点出现故障，可以快速切换到从节点作为新的主节点，从而实现故障的快速恢复。
  4. 数据分析：从节点可以用于数据分析、报表生成等读密集型任务，从而避免影响主节点的性能。
  5. 高可用性：通过主从复制，可以提高系统的可用性，当主节点发生故障时，可以快速切换到从节点继续提供服务。

2.主从关系建立

2.1 主从节点复制数据流程

  1. 主从配置好后，slave开启复制功能，slave根据配置的地址连接到master，并发送同步命令请求。

  2. master接收到同步命令后，开始执行一个后台子进程（防止请求堵塞），去进行数据快照保存与发送。

  3. master将当前实例的所有数据进行快照保存发送给从服务器。（快照保存的数据是二进制格式）

  4. slave的主进程接收到，会将这些数据加载到自身的服务中，加载后slave本地会生成持久化文件但数据是二进制的。

  5. master原数据同步后，接收到的新命令会记录到缓冲区的一个特定位置，叫做后备日志（包括读写，写请求是同步数据，读请求是读写分离），后备日志记录的是RESP格式（redis客户端与服务端通信的协议的文本格式）的命令

  6. slave主进程接收到命令后，进行执行命令同步数据，并且命令执行时，若slave也开启了持久化，那么也会进行一个持久化保存到slave节点的持久化文件中

3.3复制信息参数解释

  1. role:slave：指示该 Redis 服务器的角色是从服务器。
  2. master_host:172.16.1.51：指示当前从服务器正在复制的主服务器的 IP 地址。
  3. master_port:6379：指示当前从服务器正在复制的主服务器的端口号。
  4. master_link_status:up：指示从服务器与主服务器之间的连接状态是正常的（up），表示复制连接已经建立并且处于活动状态。
  5. master_last_io_seconds_ago:6：指示从服务器最后一次与主服务器进行通信的时间，以秒为单位。在这个例子中，从服务器在 6 秒前最后一次与主服务器进行了通信。
  6. master_sync_in_progress:0：表示当前从服务器是否正在进行同步操作。在这里，值为 0 表示不在进行同步操作。
  7. slave_repl_offset:826：指示从服务器当前的复制偏移量，即从服务器复制的主服务器上的数据偏移量。这个值可以用来检查从服务器的复制进度。
  8. slave_priority:100：指示从服务器的复制优先级。当主服务器下线后，根据优先级，会有一个从服务器被选为新的主服务器。
  9. slave_read_only:1：指示该从服务器是否处于只读模式。值为 1 表示该从服务器是只读的。
  10. connected_slaves:0：指示当前从服务器连接的从服务器数量。在这里，值为 0 表示当前从服务器没有连接其他的从服务器。
  11. master_replid:b04a24bbdd5ca5c42a1e99e619a08275683f5332：指示当前从服务器正在复制的主服务器的运行 ID。
  12. master_repl_offset:826：指示当前从服务器的复制偏移量，与 slave_repl_offset 类似，这个值也表示从服务器复制的主服务器上的数据偏移量。
  13. repl_backlog_active:1：指示是否启用了复制 backlog。值为 1 表示复制 backlog 已经启用。
  14. repl_backlog_size:1048576：指示复制 backlog 的大小，即用于保存复制操作期间的命令和数据的缓冲区大小。
  15. repl_backlog_first_byte_offset:1：指示复制 backlog 中第一个字节的偏移量。
  16. repl_backlog_histlen:826：指示复制 backlog 的历史数据的长度，即复制 backlog 中包含的历史命令和数据的长度。

4.3信息参数解释

后备日志就是master上的一个特定的缓冲区，这个缓冲区存储了master执行的命令

master会将这个缓冲区的命令发送给slave，slave进行复制数据

  1. connected_slaves:2 - 连接的从服务器数量，这表示当前主服务器连接了2个从服务器。
  2. slave0, slave1 - 从服务器的具体信息，包括IP地址、端口号、状态、复制偏移量和延迟。
    • ip=172.16.1.52,port=6379,state=online,offset=2338,lag=1
    • ip=172.16.1.53,port=6379,state=online,offset=2338,lag=1
  3. master_replid - 主服务器的复制ID，用于唯一标识主服务器。
  4. master_replid2 - 主服务器的辅助复制ID。
  5. master_repl_offset - 主服务器的复制偏移量，表示主服务器当前写入的命令的偏移量。
  6. second_repl_offset - 一个备用的复制偏移量。
  7. repl_backlog_active - 表示复制后备日志是否处于活动状态。
  8. repl_backlog_size - 复制后备日志的大小。
  9. repl_backlog_first_byte_offset - 复制后备日志的起始偏移量。
  10. repl_backlog_histlen - 复制后备日志的历史长度，即已经保存的命令数量。

8.小结

  • 主从复制功能，可以实现redis数据二次备份，master数据同步到slave

  • 并且可以实现读写分离负载均衡，使用程序代码通过负载均衡算法对slave节点进行发送读请求，降低了master压力

但是主从复制有两个问题：

  1. 单点故障：如果主库发生故障，虽然从库可以升级为主库，但是这个过程需要一定的时间，而在这段时间内整个系统将无法提供写服务。（引出redis-sentinel哨兵功能）
  2. 容量瓶颈：主从复制的架构对于处理大规模数据的场景有一定的局限性，因为所有的写操作都需要在主库上执行，主库的性能和容量可能会成为瓶颈（引出redis-cluster集群功能）
